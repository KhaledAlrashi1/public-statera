# Phase 4 — SC-3 schema DROP migrations (deploy 2)

**Status: PREPARED, HELD. Do NOT run until SC-1/2 (deploy 1) is proven live.**

SC-3 is deploy 2 of the two-deploy sequence (ruling a, 2026-07-14; see
`phase4-scope-narrowing-approvals.md`). Deploy 1 (SC-1/2) ships the code that stops
reading/writing `debt_accounts`, `savings_goals`, and `user_profiles.has_debt_choice`
while those objects still exist. SC-3 drops them **after** deploy 1 is confirmed healthy
in production (`/healthz` + `/readyz` reporting the SC-1/2 SHA, dashboards/import/export
exercised).

**Why these migration files are NOT in the SC-1/2 commit:** the production deploy runs a
one-shot `migrate` service (`docker compose run --rm migrate`). If the DROP migrations
rode along in the SC-1/2 commit, they would apply on deploy 1 — dropping the tables before
the code is proven, which is exactly what the two-deploy sequence forbids. So SC-3's
migrations are generated in their own commit in the SC-3 session.

## Migrations (three, one DDL per file — ruling a/b)

Order is fixed by ruling (a): `savings_goals` first, then `debt_accounts`, then the
`has_debt_choice` column. All three are leaf objects — no other table has an FK **to**
`debt_accounts` or `savings_goals` (verified: `savings_goals.linked_category_id` →
`categories` is `ON DELETE SET NULL` and points *away*; both tables' `user_id` FKs point
to `users`), so DROP order is safe.

1. `DROP TABLE savings_goals;`
2. `DROP TABLE debt_accounts;`
3. `ALTER TABLE user_profiles DROP COLUMN has_debt_choice;`

**Implementer's choice (ruling b): three separate one-DDL files**, generated by three
sequential `drizzle-kit generate` calls (NOT hand-written — the standing migration rules
require generate + the atomic `.sql`/journal/snapshot triple + a clean second-run check).

## Generation procedure (SC-3 session)

From `apps/api/`, one schema edit + generate per migration, in order:

1. Delete `src/db/schema/savings-goals.ts`; remove its re-export from `src/db/schema/index.ts`;
   remove the `savingsGoals` import anywhere it still lingers (should be none after SC-1/2).
   Then `pnpm --filter statera-api exec drizzle-kit generate` → produces the
   `DROP TABLE savings_goals` migration + journal entry + snapshot. Run `generate` a second
   time → must report "No schema changes."
2. Delete `src/db/schema/debt-accounts.ts`; remove its re-export from `index.ts`.
   `drizzle-kit generate` → `DROP TABLE debt_accounts`. Second-run check.
3. Remove the `hasDebtChoice` column from `src/db/schema/users.ts` (`userProfiles`).
   `drizzle-kit generate` → `ALTER TABLE user_profiles DROP COLUMN has_debt_choice`.
   Second-run check.

Then commit all three migration triples together as the SC-3 commit; deploy 2 runs
`migrate` which applies them in journal order.
