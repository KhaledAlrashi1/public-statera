### A.6 Summary

The backend is not unreachable — it is running, port 5004 is bound, and `/healthz` /
`/readyz` both return OK. The Vite `ECONNREFUSED` error the user observed was transient:
either the backend was not started at the time of the report, or it was briefly down during a
restart.

**However, there is a serious schema-migration defect that causes runtime failures on any
request that touches `memorized_transactions`.** The migration file
`b1c2d3e4f5a6_add_memorized_pin_columns.py` (added in commit `e6dd94d`) shares the same
revision ID as the pre-existing `b1c2d3e4f5a6_add_open_banking_tables.py`. Alembic warns
"Revision b1c2d3e4f5a6 is present more than once" and cannot resolve the chain, so the pin
migration was never applied. The `is_pinned` and `pinned_at` columns do not exist in the DB.
This is the root cause of Issues B and C below.

### B.5 Summary

Any import that contains at least one new transaction name (causing `learn_transaction` to
attempt a `MemorizedTransaction` SELECT) will fail with `status = "failed_internal"` on every
row. The root cause is that `memorized_transactions.is_pinned` does not exist in the database
because the migration file `b1c2d3e4f5a6_add_memorized_pin_columns.py` collides with the
pre-existing `b1c2d3e4f5a6_add_open_banking_tables.py` and was never applied. The import
validation rules themselves have not changed; the failure is a DB-level error, not a
validation rejection. PR 2's change to `learn_transaction()` (storing NULL instead of
"Uncategorized") has no downstream effect on import validation — `learn_transaction` is called
after the transaction is already written, not as a gate.

def learn_transaction(name: str, user_id: int, category: str | None = None, merchant: str | None = None) -> None:
    """Upsert memorized transaction in the current session."""
    from backend.models import MemorizedTransaction

    normalized = _txn_norm(name)
    if not normalized:
        return
    row = MemorizedTransaction.query.filter_by(norm=normalized, user_id=user_id).first()
    now = datetime.now(timezone.utc)
    real_category = category if (category and category != UNCAT_NAME) else None
    if row:
        row.count = int(row.count or 0) + 1
        row.last_seen = now
        if real_category and (not row.category or row.category == UNCAT_NAME):
            row.category = real_category
        if merchant and not row.merchant:
            row.merchant = merchant
    else:
        db.session.add(
            MemorizedTransaction(
                canonical=(name or "").strip()[:255],
                norm=normalized,
                category=real_category,

                last_seen=now,

### C.6 Summary

The probable cause ranking:

1. **Missing `is_pinned` migration (highest probability, explains all observed behavior).**
   `suggest_transactions()` fails with `ProgrammingError: column memorized_transactions.is_pinned does not exist`.
   The `useSuggestions` hook swallows the error. The dropdown shows nothing. The user cannot
   pick a suggestion and therefore cannot trigger `applyTransactionSuggestion` at all.
   The PR 2 code is correct and present; it just cannot execute.

2. **Data residue (secondary, would apply even if migration were fixed).**
   Any memorized rows written before PR 2 landed carry `category = 'Uncategorized'`. The fix
   writes new rows with `category = NULL`. If a user picks an old row (from before the fix),
   `suggestion.category` is `'Uncategorized'`, `applyTransactionSuggestion` sets the field to
   `'Uncategorized'`, and the category dropdown shows "Uncategorized" — which the user may
   perceive as "not filled." This is a data-residue issue; the code is correct.

3. **Field-name mismatch (ruled out).** Backend and frontend use the same field name
   `category`. Not the cause.

4. **Fix not present in code (ruled out).** Both `learn_transaction` and
   `applyTransactionSuggestion` carry the PR 2 changes verbatim. Not the cause.

5. **Form-level override (ruled out).** No `useEffect` or `reset()` call is triggered by a
   suggestion pick. Not the cause.

## Final section — Cross-issue observations

**Are any of the three issues related?**

Yes. Issues B and C share a single root cause: **the revision ID collision on migration
`b1c2d3e4f5a6`**, which left `is_pinned` and `pinned_at` out of the database schema while the
`MemorizedTransaction` ORM model declares those columns. Every ORM access to that table now
fails at the PostgreSQL layer.

- **Issue A (backend ECONNREFUSED)** is independent — the backend was simply not running when
  the error was observed. It is running now. It does not share a root cause with B or C.
- **Issue B (import blocked)** is caused by the missing migration: `learn_transaction()` is
  called inside the import commit path and triggers a `ProgrammingError` on the
  `MemorizedTransaction` SELECT, which is caught as `SQLAlchemyError`, marks every row as
  `failed_internal`, and causes the atomic batch to be rolled back.
- **Issue C (suggestion category not filling)** is caused by the same missing migration:
  `suggest_transactions()` fails on the same SELECT, the error is silently swallowed by the
  frontend, and the dropdown shows empty — preventing the user from ever triggering
  `applyTransactionSuggestion`.

**Could a partially-applied migration explain B or C?** Yes — this is exactly what happened.
The `is_pinned`/`pinned_at` migration was never applied due to the revision ID collision.

**Is there a single root cause that explains two or more issues?** Yes: the revision ID
collision between `b1c2d3e4f5a6_add_memorized_pin_columns.py` and the pre-existing
`b1c2d3e4f5a6_add_open_banking_tables.py` explains Issues B and C in full. Issue A is
independent (transient backend-not-running).
